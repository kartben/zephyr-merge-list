How the merge list works

This page lists pull requests that reviewers have already approved. Each one is sorted into one of three states based on the project's merge requirements.

Ready to merge now

Targets main, has no conflicts, is approved by an assignee, and has served its full review window. A release engineer can merge it right away.

Waiting on review window

Everything checks out — it just hasn't finished the mandatory waiting period yet. The row shows when it becomes eligible. Nothing for the author to do but wait.

Needs attention

Something needs a human: merge conflicts to rebase away, or a still-missing approval from an assignee. The row spells out exactly what.

The three merge requirements

  • ✓ No conflicts the branch can be rebased cleanly onto its base.
  • ✓ Assignee approved at least one assignee has approved (or the author is an assignee, or it's a hotfix).
  • ✓ Review window met the PR has been open for review long enough (see below).

Review windows

  • Normal PRs: 48 business hours (weekends don't count).
  • Trivial PRs: 4 hours.
  • Hotfix PRs: no wait — eligible immediately.

Tags you may see

  • hotfix urgent fix; skips the review window.
  • trivial low-risk change; only a 4-hour window.
  • override required merges even without fully green CI, by policy.
  • ci Nd the CI run behind this PR is N days old and may need a re-run.
  • review dismissed a change-request review was dismissed by someone other than its author.
  • dnm "do not merge" — held back intentionally.

Release phase

  • Integration: normal development — features and fixes flow into main.
  • Feature freeze: approaching a release — only changes milestoned for the upcoming release are listed.
